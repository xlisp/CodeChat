  二、近期：跑 v7，按设计验收

  不重复设计文档内容，只补三点执行建议：

  1. 跑之前先确认 stage 1 数据打印的 token 比例落在 0.6–1.6 区间（脚本会 WARN）。v6 的教训是行数比例骗人，必须看监督 token 比例。
  2. v7 训完立即跑两个 eval 定调：
    - eval_funcall：期望 pass@1 ≥ 0.80（容忍比 v6 的 0.854 略降，那是换 1:1 配比的代价）；
    - eval_mbpp_pass_at_k --k 8：这是决定 code 路线的分叉点。VERDICT ≥5% → 标准 GRPO 可用，code RL 第一次有戏；1–5% → tiered + filter；<1% → 说明 SFT
  数据再平衡也救不了，必须回预训练层。
  3. v7 的 smoke 只查 location 字段存在性，参数幻觉（Tokyo→Berlin）v7 大概率还在——设计文档 7.1 自己承认了。修法见下面第五节。

  三、中期：提升 code 能力上限（预训练层）

  这是"训练出更强代码模型"的真正答案。两条路，我推荐 A：

  方案 A — 续预训练 / mid-training（推荐）

  从 codechat_8b/latest.pt 续训（pretrain 本来就不存 optimizer state，restart 是设计内行为）：

  - 语料扩容：现在的 4B tokens 已经基本吃完一遍。目标再喂 30–80B tokens，混合：
    - the-stack-v2 Python 子集（去重后远大于现在 codeparrot 的量）
    - 20–30% 自然语言（FineWeb-Edu 之类）——没有 NL 先验，指令跟随和"理解题意"都是空中楼阁，这也解释了为什么 MBPP（自然语言题面）这么差
    - 5–10% 数学语料（OpenWebMath / proof-pile-2）——为后续加数学能力埋种子
  - 退火（annealing）阶段：最后 10–20% 步数把高质量数据（教科书式代码、MBPP/HumanEval 风格的短函数+测试文本）占比拉高、lr 退到很低。Llama 3 / MiniCPM
  都验证过这一招对下游 benchmark 的杠杆极大。
  - 代价：48.9h 训了 4B tokens，30B tokens 约 15 天 ×8 卡。贵，但这是唯一能抬上限的投资。

  方案 B — 战略收缩到 2–3B

  同等算力下，8B×4B tokens 远不如 2B×30B tokens（Chinchilla）。2B 还能甩掉 FSDP 的全部复杂度（v5 报告里两个 NCCL 去同步 bug 都是 FSDP
  带来的）。如果目标是"做出能用的模型"而非"练 8B 工程"，这条更划算。但既然 8B 管线已经跑通且 funcall 86% 证明了它能学会东西，方案 A 沉没成本更低。

  SFT 层的 code 数据补强（配合任一方案）：v7 设计 7.2 已列了 BigCodeBench、APPS；再加 OSS-Instruct（Magicoder 风格，从真实代码片段反向生成指令）和
  Evol-Instruct 类数据，比 the-stack-smol 的 module docstring 质量高得多。

  四、避免灾难性遗忘：一套可复用的规则

  把 v5/v6/v7 的教训提炼成规则，以后每加一个能力域都照此执行：

  1. 按监督 token 配比，不按行数。v6 的 5.6:1（行数）实际是 30:1（token），这是最大的坑。
  2. 任何续训阶段都带 replay：在已有能力的 ckpt 上训新域时，混入 10–30% 旧域数据。绝不在多能力 ckpt 上跑单域纯数据（v5 → funcall 专家就是这么来的）。
  3. 从中性 base 出发，不串专家 ckpt。v7 设计 §3 "不从 v6 续训" 的理由要固化成惯例：specialist 偏移逆向工程的成本高于重训。
  4. 每个域一份 held-out loss + 一个 binary smoke，训练中实时监控（v7 的 loss_code/smoke_code_pass 模式），任何一个域 loss 持续上涨立刻 kill——14h 的训练
  5 分钟能省下来。
  5. 控制 epoch 数：v6 的 7 epochs 后 5000 步都在过拟合 funcall 顺便覆写 code。混合 SFT 控制在 2–3 epochs。
  6. 消除捷径特征：v6 的 system-tag 捷径是个范式——每加新域都要问"模型能不能用一个表面 token
  模式判别这个域"，能就要做注入/负例打破它（数学域同理：如果数学样本全是 <|user|> 短问题开头，模型会学到长度捷径）。
  7. 备选的参数层手段（KL-to-base 正则、checkpoint 权重平均/model soup、LoRA-per-domain 再合并）先不用——数据层手段没榨干之前上这些是过度工程，违背仓库的
  simplicity-first 原则。

  五、多能力组合训练（code + funcall + math + …）的路线

  v7 的 source 字段 + 统一 conversations 格式天然是多域框架，加数学 = 加第四个 source，不需要新架构（mixed_sft_vs_moe.md 的结论我同意：2–5
  个相关能力域用混合 SFT，MoE 是 10+ 域才考虑的事）。具体到数学（建议作为 v8）：

  1. 前置条件：先完成第三节的 mid-training 把数学语料喂进 base。跳过这步直接 SFT 数学会重演 MBPP 的故事——base 没有先验，SFT 学到的只有格式没有推理。
  2. 数据：GSM8K train（7.5k）+ MetaMathQA（395k，做增广）。按监督 token 与 code、funcall 各 ~1:1:1 配比，cap 多余的。
  3. 监控复制 v7 模式：eval_math.jsonl held-out + 一个 smoke（"What is 17×23?" 期望 #### 391）。
  4. 数学 RL 是性价比最高的 RL：#### N 正则匹配的二元 reward 在 MathGPT 上从 0.41 起步、Pass@1 到 14.75%——同框架已验证。SFT 教会格式后 RL
  有真实梯度，这和 MBPP（恒 0）、funcall（饱和）形成对照。RL 阶段可以只跑数学，但 rollout prompt 池里混入 code/funcall prompt 做 KL 锚或定期 eval，防止
  RL 阶段单域漂移。
  5. funcall 的下一步提升（可与 v8 并行）：
    - 参数幻觉：SFT 阶段做参数值随机替换增强（见到 location: "Tokyo" 就 50% 概率换城市，v7 doc 7.1 的方案），让模型学会"从 user query
  抄参数"而不是"背高频值"；
    - 当前 122 题 eval 已被做穿（pass@16 = 0.94），要继续从 RL 拿收益必须换更难的 eval：多轮调用、多工具选择、该拒绝调用的无关工具负例（BFCL 风格）。
